# Quick Task 260701-m2r: Infer Vite Requires and Preserve in CLI

**Date:** 2026-07-01
**Status:** complete

## Scope

Implement opt-in `@napplet/vite-plugin` requirement inference for existing canonical NAP domains and make `@napplet/cli` preserve plugin-emitted `requires` tags when building deploy manifest templates.

Explicit `requires` remains supported and authoritative. Inference is tooling assistance only: no new manifest tags, domains, runtime permission policy, or shell enforcement changes.

## Tasks

1. Add Vite-plugin tests for static import inference, direct `window.napplet.<domain>` inference, type-only import ignoring, dynamic access ignoring, dedupe, warn mode, and error mode.
2. Add CLI tests proving plugin sidecar `requires` tags are preserved on root/named/snapshot deploy templates.
3. Implement a canonical-domain source scanner in the Vite plugin.
4. Normalize `requires` from existing array config or object config `{ infer, explicit, mode }`.
5. Merge explicit and inferred requirements for HTML meta and manifest tags; warn/error when inferred domains are missing from explicit config under the configured mode.
6. Read `.nip5a-manifest.json` sidecars in the CLI and copy only recognized manifest metadata tags into generated deploy templates.

## Outcome

- Added opt-in `requires: { infer: true, explicit?, mode? }` support to `@napplet/vite-plugin`.
- Implemented narrow source inference for runtime imports from `@napplet/nap/<domain>`, SDK domain subpaths, and direct `window.napplet.<domain>` access.
- Preserved existing explicit `requires: string[]` behavior.
- Ignored type-only imports and dynamic `window.napplet[domain]` access.
- Merged and deduped explicit/inferred canonical NAP domains for HTML meta and manifest `requires` tags.
- Added warn/error diagnostics when inference finds a domain missing from explicit config.
- Updated `@napplet/cli` deploy manifest generation to preserve canonical `requires` tags from plugin sidecars on root, named, and companion snapshot templates.
