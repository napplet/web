<objective>
Add a `napplet keys connect` subcommand to the Deno CLI that runs a NIP-46
remote-signer login flow (nostrconnect QR + bunker:// paste race) using
nostr-tools, then persists the result as an nbunksec in the OS keychain and
points `.napplet` signing.keyReference at it — mirroring the existing
`keys store` + `keys use` behavior.

Purpose: Let developers onboard a remote NIP-46 signer (e.g. a phone app) for
CLI signing without pasting a raw nsec, matching nsyte's bunker login ergonomics.

Output: New `packages/cli/src/nostr-connect.ts` module, wired `connect`
subcommand, updated dep maps, unit tests, docs, and a changeset.
</objective>

<non_goals>
CRITICAL PROTOCOL BOUNDARY (AGENTS.md "Protocol fidelity"): This is standard
NIP-46 remote-signer onboarding for the CLI signer. It MUST NOT touch, invent,
or depend on any NIP-5D / NAP wire surface, `napplet-*` manifest tags, the
iframe `srcdoc`/`sandbox` model, or the napplet artifact shape. NIP-46 (24133
kind, nostrconnect://, bunker://) is Nostr signer transport, entirely separate
from the napplet ⇄ shell protocol. Do not add any conformance check, build
assertion, or protocol constant. If NIP-46 needs a message the spec does not
define, STOP and flag — do not invent.

Also out of scope: replacing the existing `bunker://` "not implemented yet"
throw in signing.ts createSignerFromSecret (sign-time path); this plan only adds
onboarding that stores an nbunksec, which the existing sign-time path already
consumes.
</non_goals>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| terminal stdin → CLI | user pastes a bunker:// URL; untrusted text parsed by parseBunkerInput |
| CLI → relays (24133) | remote signer events over NIP-46; relays are untrusted transport |
| CLI → OS keychain | nbunksec secret written to native credential store |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-fb4-01 | Tampering | parseBunkerInput on stdin | mitigate | rely on nostr-tools parseBunkerInput validation; ignore non-bunker:// lines; null result loops, never throws into persistence |
| T-fb4-02 | Information disclosure | nbunksec at rest | mitigate | store only via OS keychain provider (KEY_SERVICE_NAME); never write secret to .napplet or logs; only keyReference name persisted to config |
| T-fb4-03 | Denial of service | fromURI never resolves | mitigate | 120s timeout + AbortController; pool.close + signer.close in finally so no leaked subscriptions |
| T-fb4-04 | Spoofing | rogue relay echoes secret | accept | NIP-46 secret-echo is the protocol's own auth; out of scope to harden beyond spec |
| T-fb4-SC | Tampering | jsr:@libs/qrcode install | mitigate | verify @libs/qrcode on jsr.io before pinning; render-only dep, no network/keychain access; blocking human checkpoint (Task 3 review) confirms QR output |
</threat_model>
