<objective>
Populate the NIP-5A spec-defined single-value `title` and `description` manifest
tags end to end: authored in @napplet/vite-plugin as plain HTML (`<title>` +
`<meta name="description">`), then read back out of the built index.html by the
napplet CLI at deploy time and emitted as `["title", ...]` / `["description", ...]`
manifest tags.

Purpose: These tags are already defined by NIP-5A (and inherited by NIP-5D
manifest kinds) and the CLI snapshot path already passes them through
(manifest.ts:89), but nothing currently populates them. This closes that gap.

Output:
- vite-plugin `title` / `description` options that inject/override built HTML.
- CLI deploy-time extraction of those values into title/description manifest tags.
- Unit tests in both packages, README/docs updates, and two changesets.
</objective>

<protocol_fidelity_note>
The only wire/manifest surface touched here is the NIP-5A `["title", ...]` and
`["description", ...]` single-value tags, which are spec-defined (verified by the
orchestrator against NIP-5A and the NIP-5D PR #2303). The vite-plugin change
injects a PLAIN HTML `<title>` element and a PLAIN `<meta name="description">`
element — standard HTML, NOT a `napplet-*` protocol meta tag, and NOT modeled as
protocol surface. No new envelope type, manifest field, meta name, or handshake
is introduced. If during execution any part of this design appears to require
inventing surface beyond these two spec-defined tags, STOP and flag it per
AGENTS.md rules 1–2 rather than inventing.
</protocol_fidelity_note>

    - Given `description: 'A cool napplet'` and an index.html containing `<meta name="description" content="stale">`, the transformed HTML contains the new content and NOT `stale` (single or double quotes, and name/content attribute order both handled).
    - Given `description: 'A cool napplet'` and no existing description meta, the transformed HTML gains `<meta name="description" content="A cool napplet">`.
    - Given NEITHER option, the transformed HTML preserves the author's existing `<title>`/description meta verbatim and emits no empty title/description tag; the existing napplet-type / requires / config-schema meta behavior is unchanged.
    - Attribute values in `content=` are HTML-attribute-escaped (`"` and `&` at minimum) so a title/description containing a quote does not break the tag.

  <action>
    Add optional `title?: string` and `description?: string` fields to
    `Nip5aManifestOptions` in types.ts with JSDoc: `title` sets/overrides the
    built HTML `<title>`; `description` sets/overrides `<meta name="description">`;
    both are plain HTML (NOT napplet-* protocol meta) and, when absent, leave the
    author's existing HTML untouched. Note in JSDoc that the napplet CLI reads
    these back out of the built index.html to emit NIP-5A title/description
    manifest tags.

    In html.ts add a pure exported helper (e.g. `applyHtmlMetadata(html, { title, description })`)
    that returns the modified html string. Semantics — OVERRIDE when the option
    is set, otherwise leave untouched:
      - title set + existing `<title>...</title>` present (case-insensitive,
        match across newlines) → replace inner text only.
      - title set + no `<title>` → insert `<title>...</title>` immediately after
        the opening `<head>` tag (or, if no `<head>`, prepend to the document).
      - description set + existing `<meta ... name="description" ...>` (handle
        single/double quotes and either name/content ordering, reuse the same
        attribute-matching style as the existing getAttr regex) → replace its
        `content` attribute value.
      - description set + no such meta → insert `<meta name="description" content="...">`
        after `<head>` (same fallback as title).
    Escape the injected title text for element context and the description value
    for attribute context (escape `&` and `"`; also `<`/`>` for the title inner
    text). Do NOT touch the html when both options are absent (return the input
    unchanged).

    Wire into the Vite hook: change `transformIndexHtml` in index.ts so that when
    `title` or `description` is set it returns the html-string transform form
    `{ html: applyHtmlMetadata(html, options), tags }` instead of only the tag
    array (per the design note: tag descriptors APPEND and cannot override an
    existing `<title>`/meta, so an html-string transform is required). When
    neither option is set, keep returning the existing tag array unchanged so no
    behavior regresses. Keep `buildIndexHtmlTags` responsible for the existing
    napplet-type/requires/config-schema meta tags; do not fold title/description
    into those meta tags. The vite-plugin does NOT emit title/description manifest
    tags — that is the CLI's job (Task 2).

    Add unit tests to index.test.ts asserting the six behaviors above using real
    HTML vectors (call the plugin's `transformIndexHtml` directly with an html
    string and assert on the returned `{ html }`; follow the existing test
    harness style). Include the override-existing-title, inject-when-absent,
    override-existing-description, inject-description-when-absent,
    both-absent-leaves-untouched, and quote-escaping cases.

    Update packages/vite-plugin/README.md and apps/docs/packages/vite-plugin.md:
    document the new `title` and `description` options next to `requires` /
    `archetypes`, stating they inject/override plain HTML `<title>` /
    `<meta name="description">`, are override-when-set / untouched-when-absent, and
    that the CLI turns them into NIP-5A title/description manifest tags at deploy.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| built index.html → manifest tag | Author-controlled HTML text is read and emitted as manifest tag values published to relays. |
| plugin option → built HTML | Author-supplied `title`/`description` strings are injected into the served `/index.html`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-eh0-01 | Tampering | vite-plugin `applyHtmlMetadata` | low | mitigate | HTML-escape injected title (element context: `& < >`) and description (attribute context: `& "`) so a value containing a quote/angle bracket cannot break out of the tag or inject markup. |
| T-eh0-02 | Denial of Service | CLI/plugin HTML regexes | low | mitigate | Use bounded, non-backtracking-prone regexes (first-match, character-class bounded) mirroring the existing getAttr style; parse each file once; do not build catastrophic nested-quantifier patterns over untrusted HTML. |
| T-eh0-03 | Information disclosure | title/description manifest tags | low | accept | Values are author-authored build output already intended for publication; trimming + single-value emission is sufficient. No secret-bearing surface is touched. |
</threat_model>
