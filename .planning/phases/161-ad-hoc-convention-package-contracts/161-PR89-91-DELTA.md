## Exact target semantics

| Delta | Required target semantics | Explicit non-semantics |
|---|---|---|
| Sender attestation (PR #89) | The runtime derives `inc.event.sender` from the authenticated emitting endpoint, ignores or rejects caller-supplied sender data, and documents the endpoint binding in the projection. PR #91 applies the same rule to `IntentDelivery.sender`. | No napplet-controlled sender field; no claim that `MessageEvent.source` is the universal implementation outside the web projection; no per-message signature requirement. |
| Authoritative intent URI (PRs #90/#91) | Public `intent.invoke(uri, options?)` derives `archetype`, `action`, and the queryless `convention` from `napplet:<archetype>/<intent>[...?params]`. Query pairs become a shallow text-to-text payload before `postMessage`; literal `+` stays `+`; no scalar coercion. The runtime rejects a normalized request with query/fragment in `convention` or a disagreement between URI-derived and wire `archetype`/`action`. | Do not add query handling to arbitrary domains, manifest matching, subscriptions, or payload schemas. INC remains `emit`-specific; intent gets its own invoke/open binding. |
| Rejection boundary | Fragment, malformed percent encoding, repeated decoded query name, and query plus explicit payload reject before `postMessage`. Structured/non-text payload uses a queryless URI plus `options.payload`. | Do not silently repair, normalize Unicode/case, form-decode `+`, merge payloads, or allow a caller to override URI-derived fields. |
| Acceptance, not handling (PR #91) | `IntentResult.ok === true` means the runtime accepted responsibility to deliver. Success includes normalized archetype/action/convention and resolved handler. `ok === false` is pre-acceptance rejection and carries an error. No second result reports post-acceptance delivery outcome. | Do not expose `handled`, `windowId`, a delivery ID, or imply that success means the target has already received/handled the payload. |
| Lifecycle-independent delivery | After acceptance, the runtime retains the normalized delivery independent of the source, may start/reuse/overlap/close surfaces by policy, and sends `intent.deliver { delivery }` only when the target is ready. The target receives the same carrier-neutral `IntentDelivery` whether already running or cold-started. | NAP-INTENT must not require NAP-INC. A runtime may use INC internally, but that is not target-visible contract surface. No shell lifecycle policy belongs in the napplet shim. |
| Delivery listener | `onDelivery(handler)` receives `IntentDelivery`; the runtime-provided binding retains an incoming delivery until a handler is registered. `intent.deliver` has neither request nor delivery ID. | Do not correlate delivery with `intent.invoke.id`, expose a window identity, or discard a pre-registration delivery. The draft does not state queue cardinality for multiple pre-registration deliveries; do not invent a drop/overwrite rule. |
| Handler discovery and kinds | Each manifest `archetype` tag carries a queryless convention. One tag maps to one `IntentContract { convention, eventKinds? }`; trailing `kind:<number>` values apply only to that tag. `available()`/`handlers()` expose `contracts` and may derive actions/conventions. The runtime does not inspect payload to infer kind. | Do not place query parameters in handler metadata, aggregate several conventions into one tag, infer a kind from payload, or resurrect numbered-NAP negotiation. |
| Handler selection and provenance | Runtime resolves installed manifest contracts and user default/choice. Specific handler addressing remains allowed only when the user has authorized it. Targets treat attested sender as provenance and payload as untrusted. | No client-side authorization shortcut, direct cross-napplet channel, user-default mutation, or payload trust based only on sender. |

## Required test matrix

1. **Intent URI binding:** URI derives all three wire identity fields; query mapping preserves `+`; each invalid form sends no envelope; explicit payload works only with a queryless URI.
2. **Runtime conflict rejection:** a forged/hand-built wire request with query/fragment or URI-derived field disagreement is rejected before handler resolution.
3. **Acceptance boundary:** `ok: true` has normalized fields and handler but no `handled`/`windowId`; `ok: false` has only pre-acceptance error; no post-acceptance second result.
4. **Delivery:** inbound `intent.deliver` has no IDs, reaches only `onDelivery`, is buffered before listener registration, and is identical for hot/cold target fixtures. Multiple early-delivery semantics require an upstream decision before asserting queue behavior.
5. **Sender:** INC and intent tests prove caller payload cannot select sender; the reference harness supplies endpoint identity independently. Web-projection tests may use `MessageEvent.source`, but core NAP tests must not require that carrier.
6. **Manifest discovery:** queryless convention only; same-tag optional kinds parse to unsigned event-kind metadata; repeated tag produces separate contracts; action/convention lists derive from contracts; payload is never inspected to infer kind.
7. **Carrier independence:** an intent delivery test passes with no INC domain injected and no `inc.event` sent. Existing INC exact-topic tests remain unchanged except provenance wording.
8. **Regression/documentation:** package builds/type-checks, focused Vitest/Deno suites, docs build, skills packaging test, convention-contract guard, and a repository search for `handled`, `windowId`, `intent.deliver` IDs, “only INC”, and three-element-only archetype claims.

## Scope fences

- **No source, plan, summary, changelog, or runtime-state change is made by this research artifact.**
- Do not touch `packages/cli/dist-bin/` or `workshop/`; both are unrelated untracked paths.
- Do not invent payload schemas, kind inference, a multi-contract encoding, a CLI syntax for kinds, handler authorization policy, retry/persistence policy, or multiple-delivery buffering semantics where PR #91 is silent.
- Do not make NAP-INTENT depend on NAP-INC, route an intent through a public `inc.event`, or add an INC delivery ID.
- Do not add query parsing to subscriptions, shell matching, manifests beyond rejecting query-bearing handler metadata, arbitrary domains, or already-normalized exact routes.
- Do not weaken NIP-5D web identity guidance. Move only NAP-level sender claims to carrier-neutral wording; the web projection may still document its `MessageEvent.source` binding.
- Do not alter NIP-5A aggregate hashing: archetype metadata remains outside the path-tag aggregate fold.

## Planning gates and open decisions

1. **Adoption gate:** PRs #89–#91 are open drafts. Confirm adoption/pin the exact upstream heads before changing published SDK or conformance surface.
2. **CLI/config gate:** PR #91 specifies manifest tags and `IntentContract.eventKinds`, but not the public Vite/CLI configuration spelling for kinds. Obtain an upstream/project decision rather than adding a new delimiter or flag.
3. **Delivery-buffer gate:** PR #91 requires retention before `onDelivery` registration but does not specify queue cardinality/overflow. Obtain clarification before codifying multi-delivery behavior.
4. **Conformance gate:** Decide whether the reference shell is upgraded into a lifecycle-capable runtime fixture. It must not claim sender attestation or delivery behavior it cannot model with authenticated endpoint identity.
5. **Release gate:** This changes public types and serialized manifest metadata. Re-evaluate changesets for core, nap, shim, sdk, vite-plugin, CLI, conformance, docs, and skills based on the files actually changed.
