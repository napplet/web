# Phase 159.1 PRD: CLI-first developer onboarding flow

## Problem

The current developer funnel splits one job across three unrelated entrypoints:
the SPA promotes `npx @napplet/boilerplate`, deployment requires a separately
installed Deno CLI, and agent skills require a second package-specific command.
The boilerplate generator also asks for deployment metadata that belongs in
`.napplet/config.json`. A developer can finish the visible scaffold flow without
being set up to build, verify, or deploy a napplet.

## User Story

As a napplet developer, I can install one `napplet` command, create a minimal
starter, initialize its deploy identity and metadata, install the skills my
coding agent needs, build and verify the app, and deploy it without discovering
an undocumented second workflow.

## Locked Workflow

1. Install a standalone `napplet` binary without requiring Deno. Keep the JSR
   install as an alternative for Deno users.
2. Run `napplet create my-napplet` to materialize the maintained boilerplate.
   The scaffold asks for no deployment title, named-manifest identifier, or
   archetype metadata.
3. Run `cd my-napplet && napplet init`. This creates
   `.napplet/config.json` and owns the named-manifest `d` tag, title, optional
   description, canonical NAAT archetype contracts, deploy target, relays, and
   Blossom servers.
4. Run `napplet skills install --to <agent>` to install the current
   `@napplet/skills` package through the primary CLI.
5. Ask the coding agent to build the napplet using the installed skills and the
   metadata in `.napplet/config.json`, then run the project's verification.
6. Run `napplet deploy --dry-run`, followed by `napplet deploy` when ready.

## Functional Requirements

### Standalone installation

- Build Linux x86_64/aarch64, macOS x86_64/aarch64, and Windows x86_64
  binaries from `packages/cli/src/cli.ts` in GitHub Actions.
- Publish versioned release assets and a stable latest-CLI asset route.
- Generate SHA-256 checksums. Install helpers must verify the selected binary
  before making it executable or moving it into the install directory.
- Document platform-specific installation and retain the existing
  `deno install` path.

### CLI composition

- Add `napplet create [directory]` with safe argument-array delegation to
  `@napplet/boilerplate`; preserve template override and force flags needed for
  local verification and advanced use.
- Add `napplet skills [args...]` with safe argument-array delegation to
  `@napplet/skills`; all installer targets remain available.
- Both commands must fail with a clear prerequisite message when the Node/npm
  runner needed by those package-backed operations is unavailable.
- Help and completion output must show the next command in the locked workflow.

### Deployment metadata

- Extend the backwards-compatible version-1 config with napplet metadata:
  named-manifest `d` tag/name, human title, optional description, and zero or
  more canonical NAAT archetype contracts.
- Interactive init prompts for this metadata in plain language. Non-interactive
  flags remain available for CI and scripted setup.
- Validate named `d` tags against the canonical NIP-5A-compatible constraint
  already enforced by the CLI.
- Manifest generation emits only canonical NIP-5A/NIP-5D/NAPs tags. It must not
  create a generic `type` tag. Config title/description/archetypes override
  boilerplate defaults; existing plugin metadata remains a fallback.
- Existing root deploys, workspace discovery, old configs, and explicit deploy
  selection continue to work.

### Boilerplate

- Keep `@napplet/boilerplate` as the scaffold implementation used by the CLI.
- Reduce its normal input to destination/template/force concerns. Derive the
  npm package name from the destination and leave deployment metadata to
  `napplet init`.
- Direct package usage remains documented as an advanced/underlying route, not
  the primary onboarding funnel.

### SPA and docs

- Replace the SPA's alpha-gated one-command boilerplate panel with a compact,
  ordered workflow that starts with standalone CLI installation and exposes
  copyable commands for the real sequence.
- Route hero/get-started links, root docs, package docs, tutorials, and shipped
  agent skills through the same command vocabulary.
- Keep protocol claims linked to the living NIP-5D PR, NIP-5A, and NAPs/NAAT
  sources. Do not present tool configuration as normative protocol.

## Acceptance Criteria

- A clean temporary directory can run the local CLI create command against a
  local template override, initialize metadata non-interactively, install
  skills through an injected command runner, and produce a dry-run manifest
  whose `d`, `title`, `description`, and `archetype` tags match config.
- Boilerplate unit tests prove title/d-tag/archetype prompts and flags are gone.
- CLI unit tests cover argument delegation, config normalization, wizard input,
  metadata precedence, backwards compatibility, and actionable output.
- Workflow syntax and target/asset matrices are validated; installers pass
  syntax checks and checksum-success/checksum-failure smoke tests where the host
  platform permits.
- SPA screenshots at desktop and mobile widths show the complete workflow with
  no clipped command text, overlapping controls, or inaccessible locked state.
- Repo-wide stale-funnel scans and all repository definition-of-done commands
  pass before the branch is pushed and a PR is opened.

## Canonical Sources

- NIP-5D living PR: <https://github.com/nostr-protocol/nips/pull/2303>
- NIP-5A manifest schema: <https://github.com/nostr-protocol/nips/blob/master/5A.md>
- NAP/NAAT registry: <https://github.com/napplet/naps>

## Out of Scope

- Publishing a release from the local machine or merging the PR.
- Changing NIP-5D, NIP-5A, NAP, or NAAT semantics.
- Building an agent into the CLI; the CLI installs skills and hands off to the
  developer's existing coding agent.
- Replacing the external `napplet/boilerplate` template repository in this
  phase; the package and CLI continue to materialize that maintained template.
