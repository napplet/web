# Phase 159.1: CLI-first developer onboarding flow - Context

**Gathered:** 2026-07-21
**Status:** Ready for planning
**Source:** PRD Express Path (`159.1-PRD.md`)

<domain>
## Phase Boundary

Deliver one coherent developer onboarding path in this repository: standalone
CLI delivery, CLI-composed boilerplate and skill installation, CLI-owned deploy
metadata, a simplified boilerplate generator, and matching SPA/docs/tutorial/
skill guidance. The completed phase ends with a verified, pushed branch and an
open PR.

The phase does not publish packages or binaries, change external protocol
documents, implement an agent runtime, or replace the external boilerplate
repository.

</domain>

<decisions>
## Implementation Decisions

### Workflow ownership

- The primary path is fixed as install -> `napplet create` -> `napplet init` ->
  `napplet skills` -> agent build/verify -> `napplet deploy`.
- `@napplet/boilerplate` and `@napplet/skills` remain maintained package
  implementations. The `napplet` CLI composes them so developers do not need to
  discover package-specific commands.
- Scaffold input is limited to destination/template/force concerns. Deployment
  name, title, description, and archetype metadata belong to
  `.napplet/config.json` and `napplet init`.

### Protocol-safe metadata

- "Name" means the named NIP-5D manifest `d` tag.
- "Type" is represented only by canonical NAAT `archetype` contracts. The phase
  must not invent a generic `type` field or manifest tag.
- Config metadata overrides boilerplate HTML/plugin defaults during deploy;
  existing HTML/plugin metadata remains a backwards-compatible fallback.
- Existing root deploys, workspace discovery, old version-1 configs, explicit
  selections, and current JSR installation continue to work.

### Standalone binary delivery

- GitHub Actions builds all five targets already declared by the CLI's Deno
  tasks: Linux x86_64/aarch64, macOS x86_64/aarch64, and Windows x86_64.
- Releases expose versioned assets and a stable latest-CLI asset route.
- Install helpers verify SHA-256 checksums before installation and fail closed
  on mismatch.

### CLI behavior

- Package delegation uses explicit command/argument arrays, not shell command
  strings.
- Missing Node/npm tooling produces an actionable error because the created
  Vite project and package-backed scaffold/skills operations require Node even
  though the `napplet` binary itself no longer requires Deno.
- Interactive prompts use canonical language and ordered local-project metadata
  before deployment infrastructure. Non-interactive flags cover all new fields.
- Every successful stage prints the next real command.

### SPA and documentation

- The existing blurred alpha gate remains because the warning is important; the
  installer is revealed only after explicit acknowledgement.
- The SPA preserves the original one-command shape and shows only
  `curl -fsSL https://napplet.run/install.sh | sh`.
- `napplet guide` owns the remaining create, init, skills, build, and deploy
  sequence, with relevant documentation links at each stage and a documentation
  block at the end.
- The web build hosts the checksum-verifying installers at `/install.sh` and
  `/install.ps1`; onboarding surfaces do not send users to raw GitHub URLs.
- Root docs, package docs, tutorials, and shipped skills all use the same command
  vocabulary. Direct package commands remain only as underlying/advanced routes.
- Desktop and mobile screenshots are required before completion.

### the agent's Discretion

- Exact additive `.napplet/config.json` field names and internal helper
  boundaries, provided old configs remain valid and emitted tags stay canonical.
- Exact terminal copy, spacing, icon usage, and responsive arrangement within
  the existing SPA design system.
- Whether stable release assets use a rolling release tag or an installer that
  resolves the newest versioned CLI release, provided installation is stable and
  checksum-verified.
- Atomic commit boundaries and test fixture organization.

</decisions>

<specifics>
## Specific Ideas

- Expected public command shape:
  `napplet create my-napplet`, `napplet init --name my-napplet --title "My Napplet"`,
  `napplet skills install --to codex`, `napplet deploy --dry-run`, `napplet deploy`.
- The SPA should keep installation concise; the linked terminal guide carries
  the longer sequence without turning the page into a multi-step checklist.
- Release assets should retain the existing binary names from `deno.json` so
  local and CI builds describe the same artifacts.

</specifics>

<deferred>
## Deferred Ideas

- Package/binary publication and production install verification occur only
  after merge through GitHub Actions.
- Native package-manager distribution such as Homebrew, WinGet, Scoop, apt, or
  npm global installation is future work.
- Agent invocation and prompt execution stay outside the CLI; this phase only
  installs the agent's napplet skills and documents the handoff.

</deferred>
