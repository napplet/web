## User Constraints

### Locked decisions

- The primary path is install -> `napplet create` -> `napplet init` ->
  `napplet skills` -> agent build/verify -> `napplet deploy`.
- `@napplet/boilerplate` and `@napplet/skills` remain the maintained package
  implementations; the `napplet` CLI composes them.
- Scaffold input is limited to destination/template/force concerns. Deployment
  name, title, description, and archetypes belong to `.napplet/config.json`.
- "Name" is the named NIP-5D manifest `d` tag. "Type" is represented only by
  canonical NAAT `archetype` contracts; no generic `type` tag may be invented.
- Config metadata overrides boilerplate HTML/plugin defaults during deploy;
  existing metadata remains a backwards-compatible fallback.
- Existing root deploys, workspaces, old version-1 configs, explicit selections,
  and the JSR/Deno installation remain supported.
- Build the existing five Deno targets and expose versioned plus stable CLI
  release assets with checksum verification.
- Package delegation must use explicit argument arrays and actionable missing
  prerequisite errors.
- Remove the SPA's blurred alpha gate and show the complete ordered workflow,
  verified at desktop and mobile widths.
- Root docs, package docs, tutorials, and shipped skills use the same command
  vocabulary; package-specific commands become underlying/advanced routes.

### the agent's Discretion

- Exact additive config field names and helper boundaries.
- Exact terminal copy and responsive arrangement inside the existing design system.
- Rolling release tag versus newest-version resolution for stable install assets.
- Atomic commit and test fixture organization.

### Deferred ideas

- Live package/binary publication, native package-manager distribution, and
  built-in agent execution.
- Replacing the external boilerplate repository.

## Summary

The codebase already contains every core implementation needed for this goal,
but exposes them as disconnected entrypoints. `packages/cli` has safe process
helpers, a version-1 normalized config, interactive initialization, manifest
construction, and five `deno compile` targets. `packages/boilerplate` clones the
maintained template but currently owns metadata prompts and source mutations.
`packages/skills` already has a target-aware installer. [VERIFIED: codebase]

The lowest-risk architecture is composition: add `create` and `skills` wrappers
to the Deno CLI, simplify the boilerplate generator, add an optional `metadata`
object to config, and merge canonical config tags over HTML/plugin fallbacks at
manifest construction. This leaves runtime/package boundaries intact and avoids
copying generator or skill-install logic into the binary. [VERIFIED: codebase]

Deno documents that `deno compile` emits self-contained executables and supports
all five configured targets from any host. GitHub's official artifact actions
can aggregate matrix outputs, and `gh release upload --clobber` can maintain
stable assets. [CITED: https://docs.deno.com/runtime/reference/cli/compile/]
[CITED: https://docs.github.com/en/actions/concepts/workflows-and-actions/workflow-artifacts]
[CITED: https://cli.github.com/manual/gh_release_upload]

**Primary recommendation:** Keep package implementations separate, make
`napplet` the orchestration surface, and treat `.napplet/config.json` as deploy
metadata authority with canonical-tag validation at the manifest boundary.

## Project Constraints (from AGENTS.md)

- Read living NIP-5D, NIP-5A, and NAP/NAAT sources before changing manifest or
  protocol-adjacent behavior; never invent a message, field, tag, or capability.
- Preserve user work and use a clean feature branch from current `main`.
- Move code, tests, docs, and changesets together.
- Commit green checkpoints with Conventional Commit messages and explicit paths.
- Before shipping, pass full build, type-check, recursive unit tests, docs,
  AI-slop, and diff checks; push and open a PR without another prompt.
- Publishing occurs in GitHub Actions after merge, never from the local machine.

## Canonical Findings

### Manifest metadata

- NIP-5D head `78efc118` adopts NIP-5A `path`, aggregate `x`, `server`, optional
  `title`/`description`/`source`, and defines bare-domain `requires` tags under
  napplet kinds 5129/15129/35129. [VERIFIED: GitHub PR head]
- Named napplets use kind 35129 with a `d` tag. NIP-5A constrains canonical
  named-site identifiers to lowercase letters/digits/hyphens, 1-13 characters,
  not ending in a hyphen; the CLI already enforces this pattern. [CITED:
  https://github.com/nostr-protocol/nips/blob/master/5A.md]
- The NAPs NAAT registry defines `archetype` tags as the canonical role/type
  declaration. The Vite plugin already builds these tags, while the deploy CLI
  currently drops them when reading its sidecar. [VERIFIED: codebase] [CITED:
  https://github.com/napplet/naps/blob/master/ARCHETYPES.md]
- Config metadata should be converted to tags only in `manifest.ts`, where
  canonical validation and precedence can be tested centrally. [VERIFIED:
  codebase pattern]

### Binary delivery

- `packages/cli/deno.json` already compiles exactly the supported Deno target
  matrix and bakes the required permissions into each executable. [VERIFIED:
  codebase]
- Deno explicitly supports cross-compiling all configured targets from any host,
  so a single Ubuntu matrix is sufficient. [CITED:
  https://docs.deno.com/runtime/reference/cli/compile/]
- The executable does not require Deno at runtime, but `create` and `skills`
  still need a Node package runner because they delegate to npm packages and the
  scaffold itself is a Vite/Node project. This must be stated, not hidden.
  [VERIFIED: codebase]
- The JSR workflow is the correct release trigger because `@napplet/cli` is
  intentionally excluded from npm publishing. The current npm-backed
  `github-releases.mjs` cannot release CLI versions because it gates on npm
  presence. [VERIFIED: codebase]

| Developer funnel | SPA `GetStarted.svelte` | docs/tutorials/skills | Same ordered commands everywhere |

## Architecture Patterns

### End-to-end flow

```text
install script
  -> checksum-verified standalone binary
  -> napplet create
       -> npx @napplet/boilerplate (explicit argv)
  -> napplet init
       -> .napplet/config.json (d tag + title + description + archetypes + deploy endpoints)
  -> napplet skills
       -> npx @napplet/skills (explicit argv)
  -> coding agent builds and project verify runs
  -> napplet deploy --dry-run
       -> config metadata + plugin/HTML fallback -> canonical manifest tags
  -> napplet deploy
       -> Blossom upload + relay publish
```

### Pattern 1: Thin package-command wrappers

Use the existing `runCommand(command, args)` helper. Construct arrays such as
`["--yes", "@napplet/boilerplate", target, ...flags]`; never interpolate a
shell command. Propagate stdout/stderr and return the child exit code. Test via
an injected runner rather than performing network package downloads.

### Pattern 2: Additive version-1 config

Add an optional nested metadata object and normalize every member explicitly.
`defaultConfig` must deep-copy it, `normalizeConfig` must accept its absence, and
old fixtures must remain byte-semantically valid. Do not bump config version for
an additive optional field.

### Pattern 3: Keyed metadata merge

Collect fallback HTML/plugin tags, then replace singleton keys (`title`,
`description`) with non-empty config values and replace the `archetype` group
when config explicitly provides it. Preserve `requires`, paths, servers, and
other canonical tags. Deduplicate after precedence is applied.

### Pattern 4: Matrix build, aggregate release

Compile one target per matrix job, upload each binary as a workflow artifact,
download all artifacts in a single release job, generate one `SHA256SUMS`, then
upload the complete set to both a versioned CLI release and a stable rolling CLI
release. Use `contents: write` only in the release job.

### Pattern 5: Progressive onboarding UI

The get-started section should expose all six steps in one unblocked surface.
Use a compact platform selector for install commands, one copy action per
command, and ordered step labels. On narrow screens command rows wrap inside
their own track and controls retain a stable width.

### Anti-patterns to avoid

- Duplicating the boilerplate clone/copy implementation in Deno.
- Reading `.napplet/config.json` from the Vite plugin and creating a package
  cycle between build tooling and deployment tooling.
- Emitting `type`, `napplet-type`, or other private metadata as protocol tags.
- Letting HTML title silently override explicit deploy config.
- Running five release uploads concurrently against the same release.
- Installing an unverified binary or silently falling back after checksum failure.
- Hiding most of the flow behind a modal/blur gate in the SPA.

## Common Pitfalls

### Config metadata without deploy precedence

If init writes metadata but manifest generation still reads only HTML/plugin
files, the user-visible workflow lies. Add manifest-vector tests that assert
config wins and old fallbacks still work.

### Archetype validation drift

The Vite plugin supports structured contracts. Reuse the same contract shape or
a shared validator where package boundaries permit; test protocol names and
event-kind constraints. Do not accept arbitrary nested values into tags.

### Release recursion and CLI's JSR-only status

Releases created with the repository token should not be the workflow trigger.
Trigger compilation from the Version Packages main push/manual dispatch and keep
asset creation in a dependent job. Do not route CLI through npm-gated release
logic.

### Compiled binary subprocess assumptions

The standalone binary can execute subprocesses because `--allow-run` is baked
in, but users may not have `npx` on `PATH`. Detect that failure and explain that
Node is required for scaffolding/build tooling, while other CLI commands remain
usable.

### Rolling-release replacement safety

`gh release upload --clobber` deletes an existing same-name asset before
re-upload. Upload versioned assets first and only update the rolling release
after the complete artifact/checksum set exists.

### SPA command overflow

The install URLs and Windows PowerShell command are long. Use wrapping code
tracks (`overflow-wrap: anywhere`) and separate copy controls from text so mobile
layouts do not clip or shift.

## Security Threat Model

| Threat | Severity | Mitigation |
|---|---|---|
| Shell injection through target/template/skill args | High | Explicit command and argv arrays; no shell execution |
| Tampered release binary | High | HTTPS asset source plus SHA-256 verification; fail closed |
| Unvalidated config becomes arbitrary manifest tags | High | Explicit schema normalization and canonical tag builders |
| Existing config breakage | Medium | Optional field, old-config fixtures, no version bump |
| Release job over-privilege | Medium | `contents: write` only on release job |
| HTML/config metadata ambiguity | Medium | Documented deterministic precedence with vector tests |

No database, auth session, or secret-storage schema changes are introduced.

## Recommended Plan Decomposition

1. CLI config/manifest metadata and composed command flow, with unit tests.
2. Minimal boilerplate plus binary release/install delivery, with workflow/script tests.
3. Skills and documentation funnel migration, with stale-string guards.
4. SPA onboarding implementation, responsive browser verification, full gates,
   changesets, summaries, and shipping evidence.
